# Project Simurgh Paper Accuracy Audit

**Auditor:** Claude Sonnet 4.6 (senior security-research engineer role).
**Audit date:** 2026-05-25 (this pass); prior full audit 2026-05-22.
**Source under audit:** `Papers/project-simurgh/main.tex` (current working tree, commit `356543e`).
**Verification basis:** live source tree, `npm test` run on 2026-05-25 (331/331 pass), file-by-file grep across all cited source files.
**Prior audit superseded:** 2026-05-22 audit by Claude Sonnet 4.6. This is a re-verification pass after repository reorganisation and regression-test additions.

---

## Executive Verdict

The paper is accurate in its architecture, cryptographic algorithm choices, platform claims, privacy boundary, and non-claims posture. All validation numbers (331 Node, 33 Rust, 11 .NET, 8 Swift, 383 total, 16 Linux smoke, 30 Linux security assertions, 10/10 internal audit) re-verify against the live codebase on this pass.

All three issues identified in the 2026-05-22 audit are now correctly applied and verified: (1) Section IV-B bullet 3 now uses "challenge" for the device-shield daemon proof envelope; (2) Section V-D now reads "A global nonce guard …" (not per-session); (3) Section V-B now explicitly describes two distinct canonical serialisations.

One new issue found and fixed in this pass: the Node.js test count was stale (327→331) following addition of the 4-test `nonceGuardTtlReplay.test.js` regression suite. The total test count was updated accordingly (379→383) in the abstract, contribution bullet, and Table II. All claims now verify against the live codebase.

**The paper is accurate and ready for submission.**

## Required Paper Edits

Three genuine issues found in this audit:

**1. CONTRADICTION — Section IV-B bullet 3 (daemon envelope "nonce").**

The bullet reads: "Construct a signed proof envelope containing the version, platform, session identifier, node identity hash, timestamp, **nonce**, capabilities sub-object …"

The device-shield daemon proof (ECDSA P-256) uses `challenge` for replay protection, not `nonce`. Section V-B correctly identifies the twelve daemon proof fields including `challenge` and explicitly notes that the browser-paired envelope uses `nonce`. Section IV-B bullet 3 is internally inconsistent with Section V-B.

Recommended fix: Replace "nonce" with "challenge" in Section IV-B bullet 3. The sentence should read "… node identity hash, timestamp, challenge, capabilities sub-object …" — or add a parenthetical: "challenge (in the device-shield daemon path) or nonce (in the browser-paired path)."

**2. NEEDS NARROWING — Section V-D "Each session maintains a nonce guard".**

The text reads: "Each session maintains a nonce guard: a set of all nonces seen in accepted proofs."

The implementation (`src/integrity/nonceGuard.js:3-8`) is a single global in-memory store with a 5-minute TTL, not a per-session set. The source code comment explicitly states: "We do not track per-session subdivisions." The `sessionId` parameter is accepted by `check()` but ignored.

Recommended fix: "A global nonce guard maintains a set of all nonces seen in accepted proofs, expiring entries after five minutes. The proof's own `session_id` and ECDSA signature provide session binding; the nonce guard prevents submitting the same proof twice."

**3. NEEDS NARROWING — Section V-B "same canonicalisation is implemented in the Swift and Rust daemons".**

`proofCanonicalise.js` performs recursive key sorting at all depths. `canonical_json.rs` sorts top-level keys only. These are two distinct canonicalisers for two distinct proof envelope types. The claim, as written, implies a single shared implementation.

Recommended fix: "The browser-paired proof uses the recursive key-sorted canonical serialisation implemented in `proofCanonicalise.js` (mirrored in the Swift node implementation); the device-shield daemon proof uses the top-level key-sorted serialisation of `canonicaliseDaemonPayload` (mirrored in `canonical_json.rs` for the Rust daemon). Golden-fixture interop tests lock both implementations."
