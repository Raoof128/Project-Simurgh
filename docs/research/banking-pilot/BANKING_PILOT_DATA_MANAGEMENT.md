# Banking Shield Data Management Plan

## Phase A Data Source

Phase A data is synthetic and generated by local test/smoke flows.

```json
{
  "synthetic": true,
  "human_participant": false,
  "data_source": "synthetic_test_suite"
}
```

## Storage

Runtime session state is in memory. Restarting the server clears live Phase A sessions. Evidence files under `docs/research/banking-pilot/evidence/phase-a-synthetic/` are retained as aggregate gate artifacts.

## Accepted Data

Accepted data is limited to session metadata, allowed scenario categories, timestamps, local risk categories, fixed recommendation wording, and HMAC audit-chain metadata.

## Prohibited Data

No real banking data is accepted or retained. Prohibited examples include credentials, OTPs, account identifiers, balances, payee details, transaction amounts, payment references, raw transaction text, screenshots, audio, webcam data, app names, process names, and window titles.

## Evidence Retention

The Phase A evidence pack may contain:

- Test output.
- Smoke output.
- Security audit output.
- Privacy audit output.
- Accepted report fixture.
- Rejected-attempt audit fixture.
- Sonnet sanitised payload fixture.
- Closure response fixture.

The privacy audit must fail if generated evidence contains known sensitive attack values or forbidden payload structures.

## Phase B/C Data Management

Phase B and Phase C are not implemented. Any future human dry run or pilot requires a separate data-management update and approval before code changes.
