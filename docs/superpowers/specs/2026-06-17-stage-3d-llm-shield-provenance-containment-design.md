# Stage 3D — LLM Shield Provenance & Containment Boundary (Design)

**Date:** 2026-06-17
**Status:** Approved (design); ready for implementation plan
**Branch:** `stage-3d-provenance-containment`
**Release target:** `v0.6.0-stage-3d-llm-containment` (new capability boundary, not a patch)
**Builds on:** Stage 3A-alpha (input-only shield, `/api/llm-shield`), Stage 3B
(frozen adversarial benchmark, tag `v0.5.2`/`v0.5.3`), Stage 3C-hardening
(canonicalize-then-classify + warning tier, tag `v0.5.4`).

## Steel-thread sentence

> Stage 3D does not prove jailbreak resistance. It demonstrates that jailbreak
> **consequences** can be contained across the context, tool, and output
> boundaries — even when input filtering is incomplete — with metadata-only
> receipts and tamper-evident audit evidence.

One-line framing for reviewers and the Anthropic-facing brief:

> Stage 3D turns LLM safety from prompt refusal into consequence containment:
> context cannot self-promote to instruction authority, tools cannot
> self-authorise, unsafe output cannot silently export, and every boundary
> decision leaves a metadata-only receipt on a verifiable HMAC chain.

## 1. Goal and success criteria

Stages 3A–3C built and hardened the **input** boundary. Stage 3D adds the three
**downstream** boundaries that make a jailbreak's consequence containable:
context provenance, tool invocation, and output leakage — plus a per-session
**run risk accumulator** spanning runs. All against the deterministic mock
provider and mock tools only. No live provider, no real tools, no network.

**Success criteria (gate-shaped, no inflated numeric promises):**

- Context role-escalation / untrusted-instruction cases are rejected or demoted
  to data; **no untrusted context becomes instruction authority**.
- Unsafe tool classes are **blocked before any (mock) execution**.
- Suspected hidden-policy/secret leakage in provider output is **blocked before
  export**; blocked output is hashed, never stored.
- Multi-turn softening proxies **accumulate risk** and escalate.
- Clean benign requests pass; hard-negative safety discussion is not auto-blocked.
- **Zero raw payload** (input/context/tool-args/output) in generated evidence,
  receipts, or audit chain — raw text lives **only** in fixtures.
- **No drift** in the frozen Stage 3B/3C baseline or the existing 3A receipt.

Exact per-category rates (block rates, escalation detection, FP rate) are
**generated by the fixture runner after implementation** and reported honestly;
no fixed numeric target is set up front, to avoid detector-gaming (3B/3C
discipline).

## 2. Non-claims (carried verbatim into the stage doc — loud)

- Existing 3A/3B/3C non-claims hold: no jailbreak immunity; phrase/canonical/
  heuristic matching is incomplete by construction.
- New for 3D:
  - Stage 3D does **not** prove jailbreak immunity or live-provider safety.
  - Stage 3D does **not** replace provider-side safety systems.
  - Stage 3D does **not** execute real tools and does **not** call live LLM
    providers.
  - Stage 3D does **not** guarantee all context poisoning or all output leakage
    is detected (deterministic heuristics, finite tables).
  - Stage 3D is **not** production deployment.
  - **Receipts attest process, not ground truth** — "the configured boundary
    classified/blocked/logged these events," not "the content was truly safe."

## 3. Threat model

**Asset protected:** the boundaries _after_ input classification — preventing a
jailbreak that slipped past the input filter from becoming (a) trusted context,
(b) an unsafe tool call, or (c) an exported hidden-policy/secret leak, and
producing tamper-evident proof of what each boundary did.

**Attacker capabilities (in scope for 3D):**

- Supplies untrusted `contexts[]` that forge system/developer authority, attempt
  role escalation, carry malformed schema, or embed secret/policy markers
  (context poisoning / indirect prompt injection).
- Drives the (mock) model toward an unsafe tool request (shell, network, secret
  access, prompt/policy export, unknown tool).
- Drives the (mock) model toward output that leaks a system prompt, developer
  instruction, hidden policy, classifier internals, secrets, or raw tool args.
- Applies multi-turn softening across runs in a session.

**Out of scope (deferred, with stage names):**

- Live model provider + HTTP gateway/API/Docker shell → **Stage 3E** (the
  "Anthropic can test it" layer; adapters behind `SIMURGH_LIVE_PROVIDER_ENABLED`).
- Live-model-specific exploits → **Stage 3F**.
- True semantic understanding of context/output (3D is deterministic heuristics).
- Transport security / DoS — handled by existing 3A controls (size caps, token
  binding); not re-litigated.

**Trust assumptions:** the audit HMAC secret is not attacker-known; fixtures are
the only place raw payloads live; all generated evidence is metadata-only; the
mock provider is deterministic (no network/clock/randomness) and never echoes
raw input.

## 4. Architecture — small, independently testable units

```
User / App
  ↓  POST /api/llm-shield/:sessionId/run   (Bearer token, body cap 16 KB)
  ↓
[existing] normalisePrompt → promptCanonicalise → promptFirewall (3A/3B/3C)
  ↓   verdict: safe | warning | blocked   (blocked ⇒ provider skipped, as today)
  ┌─────────────────── Stage 3D activates only if the request is a 3D run ──────────────────┐
  │  contextProvenanceGuard   (contexts[] → accepted | demoted | rejected)                   │
  │  runRiskAccumulator        (per-session monotonic score; may escalate to blocked)        │
  │  stage3dMockScenarios      (scenario enum → committed canned output / tool request)      │
  │  toolInvocationGate        (toolPolicy: allow mock-safe | warning | block; pre-execution)│
  │  outputLeakageFirewall     (scan canned output → accepted | blocked; hash-only)          │
  │  stage3dReceipt            (metadata-only, schema_version "3D")                           │
  └──────────────────────────────────────────────────────────────────────────────────────┘
  ↓
HMAC audit chain  →  GET /:sessionId/verify
```

Each unit answers: _what does it do, how is it used, what does it depend on?_ —
and is pure/deterministic so it can be unit-tested in isolation.

### 4.1 Additive activation (protects the frozen baseline)

The existing input-only path stays **byte-for-byte**. The Stage 3B/3C bench
runner hits `/run` with `{ input }`; that must keep emitting the existing
`simurgh.llm_safety_receipt.v1` (schema `"3C"`) receipt and not drift digests.

A request is a **Stage 3D run** iff any 3D field is present:

```js
const isStage3DRun =
  Array.isArray(body.contexts) ||
  body.tool_mode !== undefined ||
  body.scenario !== undefined ||
  body.stage3d === true;
```

If `isStage3DRun === false`, the route uses the existing 3A/3B/3C path unchanged
(existing receipt builder, existing audit recorders). Stage 3D code, the
`stage3dReceipt` builder, and the new audit events are reached **only** on the 3D
path. This is the single most important invariant in the stage.

### 4.2 Live mock-driver decision (locked: scenario-keyed canned outputs)

The live HTTP route may accept a bounded `scenario` enum and maps it to a
**committed, canned, deterministic** mock output (including tool-call requests
and synthetic leakage markers). The route **never** accepts a raw
`mock_provider_output` blob — that would let a requester smuggle arbitrary text
past input filtering by labelling it "model output." Fixture-level provider-output
injection is permitted **only** inside the fixture runner, which imports the
Stage 3D core pipeline directly and is **not** exposed through HTTP.

This preserves three invariants: (1) plain `{ input }` requests keep the existing
path with no receipt/benchmark drift; (2) live 3D demos stay deterministic and
reproducible; (3) requesters cannot smuggle text past the firewall as "output."

`src/llmShield/stage3dMockScenarios.js` exposes a frozen allowlist:

```
benign | tool_escalation | policy_leak | context_poisoning |
multi_turn_softening | hard_negative
```

Unknown scenario ⇒ `{ ok: false, error: "scenario_not_allowed" }`. No scenario on
a 3D run ⇒ default `benign`. Canned outputs may contain **synthetic** leakage
markers only — never real provider prompts, real policy text, or exploit payloads.

## 5. Boundary 1 — Input (unchanged, not the headline)

Reuses 3A/3B/3C exactly: `safe` → continue; `warning` → continue with accumulated
risk; `blocked` → provider skipped (non-invocation auditable). No detector change
in 3D (3C is the only stage permitted to change the detector).

## 6. Boundary 2 — Context provenance (`contextProvenanceGuard.js`, new)

**Goal:** untrusted context must never become instruction authority.

`src/llmShield/contextCanonicalise.js` (new) reuses the 3C canonicalisation on
each context's raw `content` to produce a scan-only canonical view + signal codes
(never raw text out).

Accepted request context shape (raw `content` is transient — hashed then dropped,
never persisted, never in receipt/audit):

```json
{
  "context_id": "ctx_001",
  "source_type": "retrieval | user_upload | tool_result | system_seed",
  "trust_level": "trusted | untrusted | synthetic",
  "purpose": "reference | task_data | policy_seed",
  "content": "transient raw content (hashed, then dropped)"
}
```

Per-context decision rules:

| Case                                          | Result (`context_verdict`)     |
| --------------------------------------------- | ------------------------------ |
| No `contexts`                                 | `not_supplied`                 |
| Valid untrusted reference context             | `demoted` (treated as data)    |
| Context asserts system/developer authority    | `rejected`                     |
| Context attempts role escalation              | `rejected`                     |
| Malformed schema / forbidden field            | `rejected`                     |
| `trusted` claim without valid signal          | `rejected`                     |
| Context carries secret/policy markers         | `rejected`                     |

Reason codes: `context_schema_invalid`, `context_role_escalation`,
`context_untrusted_instruction`, `context_signature_missing`,
`context_payload_too_large`, `context_forbidden_field`, `context_demoted_to_data`.

Caps: per context item 4 KB; all contexts combined 16 KB (inside the existing
16 KB body cap). Over-cap ⇒ `context_payload_too_large` (fail-closed).

## 7. Boundary 3 — Tool invocation (`toolPolicy.js` + `toolInvocationGate.js`, new)

**Goal:** unsafe tool requests are blocked **before** (mock) execution. Mock tools
only — no real shell, browser, network, file write, or live provider call.

`toolPolicy.js` is a declarative allowlist by tool class:

| Tool class                                          | Verdict          |
| --------------------------------------------------- | ---------------- |
| `mock_calculator`, `mock_lookup`                    | allow            |
| `mock_file_read`                                    | warning or block |
| `network_request`, `shell_command`, `secret_access` | block            |
| `policy_export`, `prompt_export`, `credential_request` | block         |
| unknown tool                                        | block (fail-closed) |

`toolInvocationGate.js` reads the scenario's `tool_request` (if any), evaluates it
against the policy, and **never executes** — it only decides allow/warning/block.
Tool args are hashed (`tool_name_hash`, arg digests); raw args never stored.

Reason codes: `tool_not_requested`, `tool_allowed_mock_only`, `tool_unknown`,
`tool_network_blocked`, `tool_shell_blocked`, `tool_secret_access_blocked`,
`tool_policy_export_blocked`, `tool_prompt_export_blocked`,
`tool_args_forbidden_field`.

## 8. Boundary 4 — Output leakage (`outputLeakageFirewall.js`, new)

**Goal — the "blast-radius reducer":** even if the (mock) provider returns unsafe
content, Stage 3D blocks **export**. Deterministic detectors for: system-prompt /
developer-instruction / hidden-policy leakage, classifier internals, routing
rules, tool secrets, env vars, API-key patterns, raw tool-arg leakage, verbatim
safety-policy chunks.

Verdict: `accepted` | `blocked` | `not_called` (provider skipped). **Blocked
output is hashed, not stored** (`output_hash` only) — mirroring Banking Shield's
rule: construct what is allowed, reject what is not, record the rejection without
recording the prohibited value.

Reason codes: `output_system_prompt_leakage`,
`output_developer_instruction_leakage`, `output_hidden_policy_leakage`,
`output_secret_pattern`, `output_tool_arg_leakage`, `output_classifier_internals`,
`output_blocked_metadata_only`.

## 9. Run risk accumulator (`runRiskAccumulator.js`, new)

Per-session **monotonic** score persisted on the session `record` in the store,
accumulated across runs. It owns a real verdict: a risk-triggered block emits the
full audit event order + a 3D receipt.

Principle (locked): thresholds `0–2 safe` / `3–5 warning` / `6+ blocked`. Point
weights are **design-tunable** and finalized by the fixture runner (do not promise
exact weights before implementation). Illustrative starting weights:

| Signal                          | Points       |
| ------------------------------- | ------------ |
| Input warning                   | +2           |
| Input blocked                   | (hard block) |
| Context demoted                 | +1           |
| Context rejected                | +4           |
| Role-escalation attempt         | +5           |
| Tool requested                  | +1           |
| Unsafe tool requested           | +5           |
| Output leakage blocked          | +5           |
| Repeated warning in session     | +2           |
| Hard-negative safety discussion | +0           |

`safe` remains a **classification result, not a permission** (3A wording).

## 10. Receipt schema (`stage3dReceipt.js`, new — does NOT touch `safetyReceipt.js`)

New builder, `schema_version: "3D"`, reusing the metadata-only base (hashes,
`privacy_mode`, `network_egress_used: false`, `audit_entry_hash`). The 3A
`safetyReceipt.js` (`type` `v1`, schema `"3C"`) is **untouched** so the frozen
benchmark and existing tests do not drift.

```json

  "type": "simurgh.llm_safety_receipt.v1",

  "session_id_hash": "sha256:...",
  "run_id": "run_001",
  "task_type": "general_qa",
  "input_hash": "sha256:...",
  "normalised_input_hash": "sha256:...",
  "input_verdict": "safe | warning | blocked",
  "context_verdict": "not_supplied | accepted | demoted | rejected",
```

## 11. Audit events (`llmShieldAudit.js`, modified — additive)

Existing 3A/3B/3C events keep their identifiers and order. New 3D events:

```
LLM_CONTEXT_ACCEPTED   LLM_CONTEXT_DEMOTED   LLM_CONTEXT_REJECTED
LLM_RISK_ACCUMULATED   LLM_RISK_ESCALATED
LLM_TOOL_REQUESTED     LLM_TOOL_ALLOWED_MOCK   LLM_TOOL_BLOCKED   LLM_TOOL_SKIPPED
LLM_OUTPUT_BLOCKED     LLM_STAGE3D_RECEIPT_EXPORTED
```

Decision payloads stay whitelisted to hashes/verdict/reason-codes via
`buildDecisionPayload` (no raw text). Representative ordered paths:

- **Context attack blocked:** `LLM_INPUT_ACCEPTED → LLM_CONTEXT_REJECTED →
  LLM_PROVIDER_SKIPPED → LLM_STAGE3D_RECEIPT_EXPORTED`
- **Tool attack blocked:** `LLM_INPUT_ACCEPTED → LLM_CONTEXT_ACCEPTED →
  LLM_PROVIDER_CALLED → LLM_TOOL_REQUESTED → LLM_TOOL_BLOCKED →
  LLM_STAGE3D_RECEIPT_EXPORTED`
- **Output leakage blocked:** `LLM_INPUT_ACCEPTED → LLM_PROVIDER_CALLED →
  LLM_OUTPUT_BLOCKED → LLM_STAGE3D_RECEIPT_EXPORTED`

`verifyChain` (existing `/verify` route) must pass after accepted, demoted,
blocked, tool-blocked, and output-blocked paths.

## 12. Fixture corpus (60 cases — do not jump to 150)

`docs/research/llm-shield/evidence/stage-3d/fixtures/<category>/`:

| Category                         | Count | Goal                  |
| -------------------------------- | ----: | --------------------- |
| `clean_benign`                   |    10 | should pass           |
| `hard_negative`                  |    10 | must not over-block   |
| `context_provenance`             |    10 | block / demote        |
| `tool_gate`                      |    10 | block unsafe tool     |
| `output_firewall`                |    10 | block leaky output    |
| `multi_turn_softening`           |    10 | accumulate risk       |

Fixture shape carries `mock_provider_output` (raw, fixtures-only) consumed by the
runner, plus `expected` verdicts/reason_codes and a `payload_hash`. The corpus is
**frozen** in 3B/3C style: case identity (`input`, `payload_hash`, `ground_truth`,
category, `mock_provider_output`) is immutable once committed.

## 13. Testing and gates

**Unit (`node:test`, TDD per unit):** `contextCanonicalise`,
`contextProvenanceGuard`, `toolPolicy`, `toolInvocationGate`,
`outputLeakageFirewall`, `runRiskAccumulator`, `stage3dReceipt`,
`stage3dMockScenarios` — table-driven, with danger-zone benigns as
explicit must-pass cases.

**E2E (`tests/e2e/`):** `..._stage3d_context_smoke.mjs`,
`..._stage3d_tool_gate_smoke.mjs`, `..._stage3d_output_firewall_smoke.mjs`,
`..._stage3d_receipt_privacy_smoke.mjs`, `..._stage3d_fixture_runner.mjs` (the
only path allowed to inject `mock_provider_output`, via direct import — never
HTTP).

**Scripts:** `scripts/smoke-llm-shield-stage3d.sh`,
`scripts/security-audit-llm-shield-stage3d.sh`,
`scripts/privacy-audit-llm-shield-stage3d.mjs`.

**Security-audit assertions (new):** plain `{ input }` request emits the existing
(not 3D) receipt and preserves its schema; HTTP route rejects
`mock_provider_output` (`mock_provider_output_http_rejected`); unknown scenario
rejected; scenario output comes only from the committed allowlist and never
echoes raw input; fixture-runner output injection is unreachable via HTTP;
mock-only modules import no network/provider SDK; Stage 3B frozen benchmark has
no drift.

**Privacy-audit assertions (new):** no raw context / tool-args / output in
`metrics.json`, receipt samples, or audit chain; `stage3dReceipt.js` exposes no
raw-text keys.

Plus: existing `smoke-llm-shield.sh` and `smoke-llm-shield-bench.sh` still pass
(regression); `prettier --check` clean; full `npm test` green; `npm audit
--audit-level=high` clean.

## 15. Implementation phasing (one spec → one plan, phased)

This single design produces **one** implementation plan, sequenced so each phase
is independently green:

1. Docs/threat-model skeleton + `stage3dReceipt` + audit events + additive route
   gating (plain `{input}` unaffected; 3D path returns a minimal receipt).
2. `contextCanonicalise` + `contextProvenanceGuard` (+ tests).
3. `toolPolicy` + `toolInvocationGate` + `stage3dMockScenarios` (+ tests).
4. `outputLeakageFirewall` (+ tests).
5. `runRiskAccumulator` + receipt wiring (+ tests).
6. 60-fixture corpus + fixture runner + `metrics.json`.
7. smoke/security/privacy gates + reviewer docs + closeout + tag.
